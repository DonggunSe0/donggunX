# 01. 서비스 개요

## 1. 현재 프로젝트 분석 요약

문서 작성 전 확인한 현재 프로젝트 상태는 다음과 같다.

| 항목 | 확인 내용 |
|---|---|
| 프로젝트 루트 | `/Users/apple/devCourse/demo1` |
| 빌드 도구 | Gradle |
| 현재 프로젝트명 | `demo1` |
| 현재 패키지 | `com.example.demo` |
| 현재 Java 설정 | `JavaLanguageVersion.of(17)` |
| 현재 Spring Boot 설정 | `org.springframework.boot` `4.0.6` |
| 현재 의존성 | `spring-boot-starter`, `spring-boot-starter-test`, `junit-platform-launcher` |
| 현재 백엔드 코드 | 기본 `Demo1Application` 및 기본 컨텍스트 로드 테스트만 존재 |
| 현재 설정 파일 | `src/main/resources/application.properties`에 `spring.application.name=demo1`만 존재 |

> 본 문서 세트는 구현 코드를 작성하지 않고, 이후 구현 시 기준으로 사용할 설계만 정의한다. 실제 구현 단계에서는 현재 설정을 최대한 유지하되, 포트폴리오 목표 스택인 Java 21, Spring Boot 3.x, Spring Security, JWT, Spring Data JPA, MySQL, Swagger/OpenAPI, JUnit5, Mockito, H2 또는 Testcontainers 도입 여부를 별도 작업으로 판단한다.

## 2. 서비스 이름 제안

서비스명: **PayBridge**

의미:

- 사용자와 사용자 사이의 안전한 송금을 연결하는 핀테크 백엔드 서비스
- 실제 금융망, 실명 인증망, PG 결제망은 연동하지 않고 Mock 기반으로 금융 서비스의 핵심 흐름을 학습/구현할 수 있는 포트폴리오 프로젝트
- 토스/카카오페이와 유사한 회원, 인증, 지갑, 충전, 송금, 거래내역, 정합성 관리 흐름을 백엔드 중심으로 설계

## 3. 서비스 목적

PayBridge의 목적은 실제 서비스 수준의 복잡한 금융 인프라를 모두 구현하는 것이 아니라, 백엔드 개발자로서 다음 역량을 보여줄 수 있는 포트폴리오 프로젝트를 만드는 것이다.

- Spring Security 기반 JWT 인증/인가 설계
- Refresh Token 발급, 저장, 재발급, 로그아웃 처리
- 본인 인증 상태에 따른 기능 접근 제어
- 사용자당 1개 지갑/계좌 생성 및 서버 주도 계좌번호 자동 생성
- 충전/송금/거래내역 저장 흐름 설계
- 트랜잭션 기반 잔액 정합성 보장
- 동시 요청 환경에서의 락 전략과 데드락 방지 설계
- 멱등키 기반 중복 요청 방지
- 공통 예외 응답, Swagger/OpenAPI, 테스트 시나리오 기반의 협업 가능한 백엔드 설계

## 4. 핵심 사용자

| 사용자 | 설명 | 주요 사용 기능 |
|---|---|---|
| 일반 사용자 | 회원가입 후 본인 인증을 완료하고 지갑을 사용하는 사용자 | 회원가입, 로그인, 본인 인증, 지갑 개설, 충전, 송금, 거래내역 조회 |
| 인증 미완료 사용자 | 가입은 완료했으나 본인 인증이 끝나지 않은 사용자 | 로그인, 내 정보 조회, Mock 인증 요청/확인 |
| 향후 관리자 | 운영/모니터링을 담당할 확장 사용자 | 사용자 조회, 거래 모니터링, 지갑 잠금/해제, 이상 거래 확인 |

MVP에서는 일반 사용자 기능만 구현 대상으로 하며, 관리자 기능은 권한 구조만 확장 가능하게 설계한다.

## 5. 핵심 기능 요약

| 도메인 | 핵심 기능 |
|---|---|
| Auth | 회원가입, 로그인, JWT Access Token 발급, Refresh Token 발급/재발급, 로그아웃, 내 정보 조회 |
| Verification | Mock 휴대폰 인증 코드 요청/확인, 이름/생년월일/휴대폰 번호 기반 Mock 본인 인증, 사용자 인증 상태 변경 |
| Wallet/Account | 인증 완료 사용자만 지갑 개설, 사용자당 1개 지갑 제한, 계좌번호 자동 생성, 지갑 조회, 잔액 조회, 잠금/해제 |
| Charge | Mock 충전 요청 생성, 승인/실패 처리, 충전 성공 시 잔액 증가 및 거래내역 저장, 멱등키 기반 중복 승인 방지 |
| Transfer | 사용자 간 송금, 자기 송금 차단, 잔액/상태 검증, 원자적 잔액 변경, 양측 거래내역 저장, 동시성 제어, 멱등키 적용 |
| Transaction | 내 거래내역 조회, 유형/상태/기간 필터링, 페이징, 최신순 정렬 |
| Common | 공통 응답 형식, 공통 예외 처리, Swagger/OpenAPI 문서화, 테스트 전략 |

## 6. 토스/카카오페이와 유사하지만 포트폴리오 수준에서 구현 가능한 범위

### 유사한 개념

- 사용자 계정 기반 금융 서비스
- 본인 인증 완료 후 금융 기능 사용 가능
- 사용자의 지갑/계좌 개설
- 잔액 충전 및 사용자 간 송금
- 거래내역 저장과 조회
- 송금 시 잔액 정합성과 동시성 제어
- 중복 요청 방지를 위한 멱등성 처리

### 포트폴리오 범위로 단순화하는 부분

| 실제 서비스 | 포트폴리오 설계 |
|---|---|
| 실명 인증 기관 연동 | Mock 인증 코드 및 Mock 본인 인증 |
| 은행 계좌 실명 조회/오픈뱅킹 연동 | 서버 내부 지갑/계좌번호 자동 생성 |
| PG/카드/계좌이체 결제 승인 | Mock 결제 승인 API |
| 금융보안원 수준의 보안·감사 인프라 | Spring Security, JWT, BCrypt, 민감정보 제외, 기본 감사 컬럼 |
| 원장/정산/대사 시스템 | Wallet balance와 Transaction append-only 기록으로 단순화 |
| FDS/AML/KYC 고도화 | 본인 인증 상태와 기본 금액 검증 중심 |

## 7. MVP 범위

MVP는 다음 기능을 구현 가능한 최소 단위로 정의한다.

1. 회원가입/로그인/JWT 인증
2. Refresh Token 발급 및 재발급
3. 로그아웃 시 Refresh Token 무효화
4. 내 정보 조회
5. Mock 휴대폰 인증 및 Mock 본인 인증
6. 인증 완료 사용자만 지갑 개설
7. 사용자당 지갑 1개 제한
8. 계좌번호 자동 생성
9. 내 지갑/잔액 조회
10. Mock 충전 요청/승인/실패
11. 사용자 간 송금
12. 거래내역 저장 및 조회
13. 송금/충전 승인에 대한 멱등키 처리
14. 트랜잭션 및 락 기반 잔액 정합성 보장
15. 공통 예외 처리 및 Swagger 문서화
16. 핵심 단위/통합 테스트

## 8. MVP 제외 범위

- 실제 금융기관 API 연동
- 실제 본인 인증 API 연동
- 실제 PG 결제 API 연동
- 외부 은행 계좌 연결
- 정산/대사/회계 원장 분리
- 관리자 화면
- 프론트엔드 구현 또는 수정
- Redis 기반 토큰/락 저장소
- 메시지 큐 기반 비동기 처리
- 실시간 알림, 푸시, SMS 발송

## 10. 전체 서비스 흐름 요약

### 10.1 가입부터 지갑 개설까지

1. 사용자가 이메일, 비밀번호, 이름, 휴대폰 번호, 생년월일로 회원가입한다.
2. 서버는 이메일 중복 여부를 확인하고 비밀번호를 BCrypt로 암호화해 저장한다.
3. 사용자가 로그인하면 Access Token과 Refresh Token을 발급한다.
4. 사용자는 휴대폰 인증 코드를 요청한다.
5. 서버는 Mock 인증 코드를 생성하고 만료 시간을 저장한다.
6. 사용자는 인증 코드를 확인한다.
7. 사용자는 이름, 생년월일, 휴대폰 번호 기반 Mock 본인 인증을 요청한다.
8. 인증 성공 시 `User.verified=true`로 변경한다.
9. 인증 완료 사용자는 지갑 개설을 요청한다.
10. 서버는 사용자당 지갑이 없는지 확인하고 계좌번호를 자동 생성한 뒤 초기 잔액 0원으로 지갑을 생성한다.

### 10.2 충전 흐름

1. 인증 완료 사용자가 충전 요청을 생성한다.
2. 서버는 충전 금액이 0원 초과인지 검증하고 `Charge`를 `PENDING` 상태로 저장한다.
3. 사용자는 Mock 결제 승인 API를 호출한다.
4. 서버는 `Idempotency-Key`를 검증한다.
5. 서버는 하나의 트랜잭션에서 지갑 잔액 증가, `Charge.SUCCESS` 변경, `Transaction.CHARGE/SUCCESS` 저장을 처리한다.
6. 동일한 승인 요청이 재시도되면 저장된 멱등 응답을 반환하거나 충돌을 차단한다.

### 10.3 송금 흐름

1. 송금자가 받는 사람 계좌번호와 금액, 메모를 입력한다.
2. 서버는 인증, 본인 리소스, 송금자/수신자 지갑 상태, 금액, 자기 송금 여부를 검증한다.
3. 서버는 `Idempotency-Key`를 검증한다.
4. 서버는 송금자와 수신자 지갑을 정해진 순서로 잠근다.
5. 하나의 트랜잭션에서 송금자 잔액 차감, 수신자 잔액 증가, 양측 거래내역 저장을 처리한다.
6. 성공 시 `TRANSFER_SEND`, `TRANSFER_RECEIVE` 거래내역이 각각 저장된다.
7. 실패 시 잔액 변경 없이 적절한 예외 응답 또는 실패 거래 상태를 기록한다.

### 10.4 거래내역 조회 흐름

1. 사용자는 인증 토큰으로 내 거래내역을 조회한다.
2. 서버는 로그인 사용자 지갑 기준으로 거래내역을 필터링한다.
3. 거래 유형, 거래 상태, 기간, 페이지, 크기 조건을 적용한다.
4. 최신순으로 페이징 결과를 반환한다.
