# 07. 동시성 및 트랜잭션 설계

## 1. 설계 목표

충전과 송금은 잔액을 변경하는 핵심 기능이다. 따라서 다음 목표를 만족해야 한다.

- 잔액이 음수가 되지 않는다.
- 송금자 잔액 차감과 수신자 잔액 증가는 항상 함께 성공하거나 함께 실패한다.
- 동시에 여러 송금 요청이 들어와도 최종 잔액이 정확해야 한다.
- 동일 요청 재시도로 중복 충전/중복 송금이 발생하지 않아야 한다.
- 실패 시 부분 반영된 데이터가 남지 않아야 한다.

## 2. 송금 로직의 트랜잭션 범위

송금 API `POST /api/transfers`의 트랜잭션 범위는 다음 작업을 포함한다.

1. 멱등키 조회/생성 및 `PROCESSING` 상태 기록
2. 송금자 지갑 조회 및 락 획득
3. 수신자 지갑 조회 및 락 획득
4. 송금자/수신자 지갑 상태 검증
5. 자기 송금 여부 검증
6. 송금 금액 검증
7. 송금자 잔액 부족 여부 검증
8. 송금자 잔액 차감
9. 수신자 잔액 증가
10. 송금자 거래내역 `TRANSFER_SEND/SUCCESS` 저장
11. 수신자 거래내역 `TRANSFER_RECEIVE/SUCCESS` 저장
12. 멱등키 `SUCCESS` 및 응답 본문 저장

위 작업 중 하나라도 실패하면 전체 트랜잭션을 롤백한다.

## 3. 잔액 차감과 증가를 하나의 트랜잭션으로 처리하는 이유

송금은 두 지갑의 잔액이 동시에 변경되는 작업이다.

- 송금자 잔액만 차감되고 수신자 잔액 증가가 실패하면 돈이 사라진다.
- 수신자 잔액만 증가하고 송금자 잔액 차감이 실패하면 돈이 생성된다.
- 거래내역만 저장되고 잔액 변경이 실패하면 조회 데이터와 실제 잔액이 불일치한다.

따라서 송금은 반드시 하나의 DB 트랜잭션에서 원자적으로 처리해야 한다.

## 4. 비관적 락과 낙관적 락 비교

| 구분 | 비관적 락 | 낙관적 락 |
|---|---|---|
| 개념 | 데이터를 읽을 때부터 다른 트랜잭션의 변경을 막음 | version 비교로 커밋 시 충돌 감지 |
| JPA 예시 | `@Lock(PESSIMISTIC_WRITE)` | `@Version` |
| 장점 | 잔액 정합성 보장이 직관적, 충돌 시 즉시 대기/차단 | 락 대기 시간이 적고 읽기 성능 유리 |
| 단점 | 락 대기, 데드락 가능성, 처리량 저하 가능 | 충돌 시 재시도 로직 필요, 고경합에서 실패 증가 |
| 적합한 경우 | 잔액처럼 충돌 시 손실 위험이 큰 데이터 | 충돌 가능성이 낮은 일반 수정 데이터 |

## 5. 이 프로젝트에서 선택할 락 전략

MVP에서는 **비관적 락**을 기본 전략으로 선택한다.

선택 이유:

- 송금은 금융 도메인의 핵심 잔액 변경 작업이다.
- 동시에 여러 요청이 들어왔을 때 잔액 부족 판단이 반드시 최신 값 기준이어야 한다.
- 포트폴리오 프로젝트에서 정합성을 설명하고 테스트하기 쉽다.
- MySQL InnoDB의 row-level lock과 JPA `PESSIMISTIC_WRITE`로 구현 가능하다.

단, `wallets.version` 컬럼은 유지한다. 이유는 다음과 같다.

- 향후 낙관적 락으로 전환하거나 혼합 전략을 적용할 수 있다.
- 변경 버전을 추적할 수 있다.
- 테스트에서 변경 횟수를 확인할 수 있다.

## 6. 잔액이 음수가 되지 않도록 하는 방법

다중 방어 전략을 적용한다.

1. 요청 검증: `amount > 0`
2. 트랜잭션 내부에서 송금자 지갑 락 획득
3. 락 획득 후 최신 잔액으로 `balance >= amount` 검증
4. 검증 통과 시에만 `balance = balance - amount`
5. DB 체크 제약 `balance >= 0` 추가 고려
6. 예외 발생 시 트랜잭션 롤백

주의:

- 락 획득 전 읽은 잔액으로 부족 여부를 판단하면 동시성 문제가 생길 수 있다.
- 반드시 락 획득 후 최신 잔액으로 다시 검증한다.

## 7. 동시에 여러 송금 요청이 들어왔을 때 처리 흐름

예시: A 지갑 잔액 10,000원. 동시에 8,000원 송금 요청 2건 발생.

1. 요청 1과 요청 2가 거의 동시에 들어온다.
2. 요청 1이 A 지갑에 대한 비관적 락을 먼저 획득한다.
3. 요청 2는 A 지갑 락을 기다린다.
4. 요청 1은 잔액 10,000원을 확인하고 8,000원을 차감해 잔액 2,000원으로 만든 뒤 커밋한다.
5. 요청 2가 락을 획득한다.
6. 요청 2는 최신 잔액 2,000원을 확인한다.
7. 요청 2의 송금 금액 8,000원이 잔액보다 크므로 `INSUFFICIENT_BALANCE`로 실패한다.
8. 최종 잔액은 2,000원으로 정합성이 유지된다.

## 8. 데드락 방지를 위한 계좌 락 획득 순서

송금은 두 지갑을 동시에 잠근다. 서로 반대 방향 송금이 동시에 발생하면 데드락 위험이 있다.

예시:

- 요청 1: A → B
- 요청 2: B → A

잘못된 방식:

- 요청 1이 A 락 획득 후 B 락 대기
- 요청 2가 B 락 획득 후 A 락 대기
- 데드락 발생

해결 방식:

- 송금 방향과 무관하게 지갑 ID 오름차순으로 락을 획득한다.
- 예: `min(senderWalletId, receiverWalletId)`를 먼저 잠그고, 그 다음 `max(...)`를 잠근다.

처리 흐름:

1. senderWallet과 receiverWallet을 조회한다.
2. 두 지갑 ID를 비교한다.
3. 작은 ID 지갑부터 `PESSIMISTIC_WRITE` 락으로 조회한다.
4. 큰 ID 지갑을 `PESSIMISTIC_WRITE` 락으로 조회한다.
5. 락 획득 후 원래 sender/receiver 역할에 맞게 잔액을 변경한다.

## 9. 멱등키를 활용한 중복 요청 방지

### 9.1 적용 대상

- `POST /api/charges/{chargeId}/approve`
- `POST /api/transfers`

### 9.2 저장 전략

`idempotency_keys` 테이블에 다음 정보를 저장한다.

- userId
- idempotencyKey
- apiPath
- requestHash
- responseBody
- status
- expiresAt

### 9.3 처리 흐름

1. 요청 헤더에서 `Idempotency-Key`를 읽는다.
2. 요청 본문을 정규화하고 hash를 계산한다.
3. `(userId, apiPath, idempotencyKey)`로 기존 레코드를 조회한다.
4. 기존 레코드가 없으면 `PROCESSING`으로 생성한다.
5. 기존 레코드가 있고 requestHash가 다르면 `IDEMPOTENCY_KEY_CONFLICT`를 반환한다.
6. 기존 레코드가 있고 status가 `SUCCESS`이면 저장된 responseBody를 반환한다.
7. 기존 레코드가 있고 status가 `PROCESSING`이면 `DUPLICATE_REQUEST_PROCESSING`을 반환한다.
8. 비즈니스 처리가 성공하면 responseBody와 `SUCCESS` 상태를 저장한다.
9. 비즈니스 처리가 실패하면 실패 유형에 따라 `FAILED`로 변경하거나 레코드를 삭제한다.

### 9.4 멱등키와 트랜잭션 경계

- 멱등키 생성과 비즈니스 처리는 가능한 같은 트랜잭션에서 처리한다.
- 다만 실패 응답까지 멱등하게 저장할지 여부는 정책이 필요하다.
- MVP에서는 성공 응답만 저장하고, 검증 실패는 저장하지 않는 방식을 권장한다.
- 처리 중 서버 오류로 롤백되면 멱등키도 롤백되어 재시도 가능하다.

## 10. 실패 시 거래 상태 처리 방식

### 10.1 송금 실패

| 실패 위치 | 거래내역 저장 여부 | 처리 방식 |
|---|---|---|
| 입력값 검증 실패 | 저장하지 않음 | `INVALID_AMOUNT` 등 예외 반환 |
| 권한/인증 실패 | 저장하지 않음 | 인증/인가 예외 반환 |
| 지갑 없음/잠김 | 저장하지 않음 | 도메인 예외 반환 |
| 잔액 부족 | 선택 | MVP에서는 저장하지 않음. 확장 시 실패 거래 기록 가능 |
| 잔액 변경 후 오류 | 롤백 | 트랜잭션 롤백으로 거래내역도 저장되지 않음 |

MVP에서는 성공한 송금만 거래내역에 저장한다. 실패 거래 감사가 필요하면 별도 `TransactionStatus.FAILED` 기록 정책을 확장한다.

### 10.2 충전 실패

| 실패 위치 | Charge 상태 | Transaction 저장 |
|---|---|---|
| 충전 요청 생성 전 검증 실패 | 생성 안 함 | 저장 안 함 |
| Mock 승인 실패 | `FAILED` | 저장 안 함 또는 `FAILED` 저장 선택 |
| 승인 중 지갑 잠금 | `PENDING` 유지 또는 `FAILED` | 저장 안 함 |
| 승인 후 오류 | 롤백 | 저장 안 함 |

MVP에서는 Mock 승인 실패가 명시적으로 발생한 경우 `Charge.FAILED`로 변경하고, 잔액은 변경하지 않는다.

## 11. 충전 승인 시 트랜잭션 처리 방식

충전 승인 API는 하나의 트랜잭션에서 다음을 처리한다.

1. 멱등키 검증/생성
2. Charge 조회 및 소유자 검증
3. Charge 상태가 `PENDING`인지 검증
4. Wallet 비관적 락 획득
5. Wallet 상태가 `ACTIVE`인지 검증
6. Wallet 잔액 증가
7. Charge 상태를 `SUCCESS`로 변경
8. `approvedAt` 저장
9. `Transaction.CHARGE/SUCCESS` 저장
10. 멱등키 응답 저장

중복 승인 방지:

- Charge 상태가 이미 `SUCCESS`이면 다시 잔액을 증가시키지 않는다.
- 같은 멱등키/같은 요청이면 기존 성공 응답을 반환한다.
- 다른 멱등키라도 이미 처리된 Charge이면 `CHARGE_ALREADY_PROCESSED`를 반환한다.

## 12. 트랜잭션 격리 수준

MVP에서는 MySQL InnoDB 기본 격리 수준인 `REPEATABLE READ`를 사용할 수 있다.

- 잔액 변경 대상 row에 비관적 write lock을 사용하므로 lost update를 방지할 수 있다.
- 단순 조회 API는 별도 락을 사용하지 않는다.
- 거래내역 조회는 최종 커밋된 데이터 기준으로 조회한다.

필요 시 송금/충전 승인 트랜잭션에 timeout을 설정해 락 대기가 무한정 지속되지 않도록 한다.
